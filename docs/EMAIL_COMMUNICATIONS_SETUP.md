# Запуск почтового контура на проде (Яндекс.Почта)

Дата: 2026-05-12.

Связанные документы:

- `docs/improvement plans/email-communications-plan.md` — продуктовая постановка;
- `docs/improvement plans/email-communications-technical-plan.md` — технический план MVP (особенно §22 — пересмотренные требования);
- `docs/DEPLOYMENT_QUICK_REFERENCE.md` — общий деплой проекта;
- `docs/ENVIRONMENT_VARIABLES.md` — справочник env.

## 1. Что уже реализовано

После двух коммитов (`feat: implement email communications MVP` и `refactor: harden email communications MVP post-review`) в системе есть:

- приложение `apps.communications` с моделями `MailAccount`, `OutboundEmail`, `OutboundEmailRecipient`, `OutboundEmailAttachment`, `EmailDeliveryAttempt`;
- SMTP-провайдер;
- Celery-задача отправки;
- сервисы `create_outbound_email`, `queue_outbound_email`, `retry_outbound_email`, `send_outbound_email_now`;
- интеграция с `BillingTask`: формы «Отправить в СК» и «Отправить в Альянс» в карточке очередного взноса;
- защита от отката статуса задачи, защита от двойного клика, retry failed-писем (UI + admin action);
- разделение доступа: feature flag `COMMUNICATIONS_RESTRICT_TO_SUPERUSER` плюс Django permission `communications.send_outbound_email`;
- 47 тестов на эту функциональность.

Чего пока **нет** (это запланировано на следующие этапы):

- автоматический подбор email СК и Альянса (адрес вводится вручную);
- четверговый отчёт через communications (этап 4 из плана);
- входящий IMAP-контур (этапы 5+);
- сохранение копии письма в папку «Отправленные» почтового ящика;
- автоматический ретрай transient-ошибок Celery.

## 2. Что нужно подготовить до начала

### 2.1 Почтовый ящик

Выберите один из двух вариантов:

**Вариант A — отдельный Яндекс-ящик (рекомендуется для теста).** Регистрируйте новый email вида `onlinepolis-billing@yandex.ru` или `op-billing@yandex.ru`. Не используйте личный ящик сотрудника — позже придётся отдавать пароль приложений на сервер.

**Вариант B — ящик в Яндекс 360 для бизнеса.** Если у компании уже есть подписка Яндекс 360, можно завести там почтовый ящик на корпоративном домене, например `billing@<ваш-домен>.ru`. Это сразу даёт DKIM/SPF/DMARC по корпоративному домену и более высокие лимиты на отправку.

Для production ставьте именно **Вариант B**, чтобы не сжечь обычный Яндекс-ящик лимитом и не попасть в спам. Для staging-теста подойдёт и **Вариант A**.

### 2.2 Включение IMAP/SMTP и пароль приложения

В настройках Яндекс-ящика:

1. Откройте «Все настройки» → «Почтовые программы».
2. Включите «С сервера imap.yandex.ru по протоколу IMAP» и «Авторизация с помощью SMTP» (или эквивалентный пункт в текущем интерфейсе).
3. Зайдите в «Управление аккаунтом» → «Безопасность» → «Пароли приложений».
4. Создайте новый пароль приложения с типом «Почта», скопируйте его (16 символов без пробелов). Этот пароль больше не отобразится.

Полученный пароль приложения и есть `COMMUNICATIONS_SMTP_PASSWORD`. Обычный пароль аккаунта **не подходит** — Яндекс блокирует SMTP-вход по основному паролю при включённой 2FA.

### 2.3 Тестовые получатели

Заведите два-три тестовых email, на которые вы будете отправлять первые письма из системы. Это могут быть ваши личные ящики на других провайдерах (Gmail, Mail.ru), чтобы проверить попадает ли письмо в инбокс / спам.

### 2.4 Что должно быть на сервере

- Docker и `docker-compose` v1 (см. CLAUDE.md — production использует `docker-compose`, не `docker compose`);
- запущенные сервисы из `docker-compose.prod.yml`: `web`, `db`, `redis`, `celery_worker`, `celery_beat`;
- Redis — потому что Celery держит очередь там;
- работающий nginx + SSL (письма не зависят, но они нужны для админки).

Celery worker и beat обязательны: без worker'а письма не уйдут (`send_outbound_email.delay(...)` поставит задачу, но её никто не возьмёт).

## 3. Настройка переменных окружения

Все переменные читает `config/settings.py`. На production они должны лежать в `.env.prod` на сервере (deploy preserves `.env.prod*`).

### 3.1 Обязательные

```env
# Главный рубильник. Пока False — система НЕ отправляет письма даже если суперпользователь нажмёт кнопку.
COMMUNICATIONS_EMAIL_ENABLED=True

# Код default-аккаунта в БД. Менять не нужно, если не планируете несколько ящиков.
COMMUNICATIONS_DEFAULT_ACCOUNT=billing

# SMTP-настройки Яндекса. SSL на 465 — самый простой вариант, не требует STARTTLS.
COMMUNICATIONS_SMTP_HOST=smtp.yandex.ru
COMMUNICATIONS_SMTP_PORT=465
COMMUNICATIONS_SMTP_USE_SSL=True
COMMUNICATIONS_SMTP_USE_TLS=False

# Логин и пароль ящика-отправителя.
# Логин — полный email; пароль — тот самый «пароль приложения» из шага 2.2.
COMMUNICATIONS_SMTP_USERNAME=onlinepolis-billing@yandex.ru
COMMUNICATIONS_SMTP_PASSWORD=xxxxxxxxxxxxxxxx

# Адрес и видимое имя отправителя. From-адрес должен совпадать с SMTP_USERNAME,
# иначе Яндекс отклонит письмо ошибкой «From address not allowed».
COMMUNICATIONS_FROM_EMAIL=onlinepolis-billing@yandex.ru
COMMUNICATIONS_FROM_NAME=ОнлайнПолис · Очередные взносы
```

### 3.2 Опциональные

```env
# Домен для Message-ID. Имеет смысл выставить совпадающим с FROM_EMAIL-доменом.
# Если не задано — будет yandex.ru, что для бесплатного ящика норм, но для прода
# на собственном домене лучше явно прописать.
COMMUNICATIONS_MESSAGE_ID_DOMAIN=yandex.ru

# Лимит размера вложения, MB. По умолчанию 10.
COMMUNICATIONS_ATTACHMENT_MAX_SIZE_MB=15

# Таймаут SMTP-соединения, сек. По умолчанию 30.
COMMUNICATIONS_SEND_TIMEOUT=30

# Пока тестирование — оставить True. Тогда отправлять может только суперпользователь,
# даже если permission выдан другим пользователям. Снять, когда тестирование закончено.
COMMUNICATIONS_RESTRICT_TO_SUPERUSER=True

# Автоматически менять статус BillingTask после успешной отправки письма.
# По умолчанию True (production-режим). Для первых staging-тестов рекомендуется
# выставить False — тогда отправка работает, история пишется, но статус задачи
# не двигается. Это позволяет проверить SMTP, заголовки, доставку и спам-фильтры
# несколько раз подряд на одной и той же задаче без последующего ручного отката.
BILLING_AUTO_UPDATE_TASK_ON_EMAIL_SENT=False

# Скрытая копия (BCC) для аудита. Адреса перечисляются через запятую или
# точку с запятой. Каждое исходящее письмо автоматически отправляется ещё
# и на эти ящики. В заголовках письма BCC получатели не видны (это норма
# для BCC). В UI карточки задачи BCC тоже не показывается — он виден только
# в /admin/communications/outboundemail/.
COMMUNICATIONS_BCC_EMAILS=mdsk95@gmail.com
```

### 3.3 Чего НЕ нужно делать

- Не выкладывать `COMMUNICATIONS_SMTP_PASSWORD` в репозиторий или в `.env.example`.
- Не использовать порт 25 — на нём почти везде запрещена авторизация.
- Не путать `EMAIL_*` и `COMMUNICATIONS_*`. Старые `EMAIL_*` остались для отдельных Django-уведомлений и не используются почтовым контуром.

## 4. Деплой

После того как `.env.prod` обновлён, делаем стандартный деплой.

### 4.1 Если деплоите через GitHub Actions

1. Слейте `main` (после моих коммитов он уже содержит весь нужный код).
2. Workflow `.github/workflows/deploy.yml` сам подключится по SSH, обновит код на сервере, применит миграции, соберёт статику и перезапустит контейнеры.
3. Дождитесь зелёного статуса.

### 4.2 Если деплоите вручную

На сервере:

```bash
cd /path/to/project
git fetch origin main
git reset --hard origin/main
docker-compose -f docker-compose.prod.yml pull
docker-compose -f docker-compose.prod.yml up -d --build
docker-compose -f docker-compose.prod.yml exec web python manage.py migrate
docker-compose -f docker-compose.prod.yml exec web python manage.py collectstatic --noinput
```

### 4.3 Что должна сделать миграция

Применятся `apps.communications.migrations.0001_initial` и `0002_alter_outboundemail_options`. Первая создаст 5 таблиц, вторая — добавит permission `communications.send_outbound_email`.

Проверка:

```bash
docker-compose -f docker-compose.prod.yml exec web python manage.py showmigrations communications
```

Обе миграции должны быть с галочкой `[X]`.

## 5. Первая настройка после деплоя

### 5.1 Создать `MailAccount`

Делать руками **не нужно**: при первом обращении `get_default_mail_account()` создаст запись `code=billing` с полями из settings (`COMMUNICATIONS_FROM_EMAIL`, `COMMUNICATIONS_FROM_NAME`). Просто откройте карточку любой задачи billing — запись появится сама.

Хотите проверить заранее:

```bash
docker-compose -f docker-compose.prod.yml exec web python manage.py shell -c \
  "from apps.communications.services import get_default_mail_account; print(get_default_mail_account())"
```

Ожидаемый вывод: `Системный почтовый ящик <onlinepolis-billing@yandex.ru>`.

### 5.2 Раздать права

Пока `COMMUNICATIONS_RESTRICT_TO_SUPERUSER=True`, нажать «Отправить в СК / в Альянс» может только суперпользователь. Это специально, чтобы на старте тестировал один человек.

Когда захотите расширить доступ:

1. В админке создайте/выберите группу «Может отправлять письма».
2. Дайте этой группе permission `communications | Исходящее письмо | Может отправлять письма из системы` (`send_outbound_email`).
3. Добавьте в группу нужных пользователей.
4. Снимите флаг: на сервере выставьте `COMMUNICATIONS_RESTRICT_TO_SUPERUSER=False` в `.env.prod` и перезапустите `web`/`celery_worker` (чтобы settings перечитались).

Только после этого участники группы увидят формы отправки и кнопку retry.

### 5.3 Проверить, что Celery работает

```bash
docker-compose -f docker-compose.prod.yml logs --tail=50 celery_worker
```

В выводе должно быть `celery@worker ready` и ни одной строки `ERROR` про подключение к Redis или брокеру. Если worker не запущен, никакая отправка не пойдёт: письмо будет вечно висеть в статусе `queued`.

## 6. Как пользоваться (UI)

### 6.1 Карточка задачи очередного взноса

Откройте `/policies/payments/scheduled/`, выберите период, кликните на задачу. На странице две панели справа: «Текст письма в [СК]» и «Текст письма в Альянс-лизинг».

Для каждой панели:

- старые кнопки «Скопировать тему» / «Скопировать текст» остаются — их видят все авторизованные пользователи (fallback на ручную отправку);
- ниже, под кнопками копирования, для тех, кому разрешено, появляется форма отправки.

### 6.2 Отправка в СК

1. Введите email получателя в поле «Email получателя». Можно указать **несколько адресов через запятую или точку с запятой** — например `ivan@sk.ru, office@sk.ru`. Дубликаты в списке вычищаются автоматически.
2. Нажмите «Отправить в СК».
3. Кнопка заблокируется, пока запрос идёт; вы увидите сообщение «Письмо в СК поставлено в очередь отправки».
4. Через 1–3 секунды (зависит от Celery + SMTP) статус задачи поменяется на «Счёт запрошен у СК», а ниже появится блок «История отправок из системы» с записью.

Если в `.env.prod` задан `COMMUNICATIONS_BCC_EMAILS`, эти адреса автоматически добавляются как `Bcc:` к каждому письму. Получатели в `To:` их не видят — это стандартное поведение BCC. В блоке «История отправок» BCC тоже скрыт; полный список получателей виден только в `/admin/communications/outboundemail/` для аудита.

### 6.3 Отправка в Альянс

1. Введите email Альянса (тоже допустимо несколько через запятую).
2. Прикрепите файл счёта (PDF / JPG / PNG / DOC / DOCX / XLS / XLSX, до 10 MB по умолчанию).
3. Нажмите «Отправить в Альянс».
4. Без файла отправка невозможна — система отклонит запрос с сообщением «Для письма в Альянс необходимо приложить счет».
5. После успеха статус задачи перейдёт в «Передан в оплату в Альянс».

### 6.4 История отправок

В блоке внизу правой колонки видны все письма, которые система отправляла по этой задаче: тип, статус, получатель, время, тема, и (через `<details>`) тело письма. Для писем со статусом «Ошибка» под каждым появится кнопка «Повторить отправку» — она отправит тот же snapshot повторно (то есть тема и тело не меняются).

### 6.5 Защита от двойного клика

После первого клика кнопка блокируется. Если по какой-то причине пользователь успеет отправить второй POST в течение 60 секунд по той же задаче и тому же типу — сервер ответит «Аналогичное письмо только что уже было создано. Подождите минуту или откройте историю отправок».

### 6.6 Режим тестирования отправки без авто-смены статуса

Когда `BILLING_AUTO_UPDATE_TASK_ON_EMAIL_SENT=False`:

- кнопки «Отправить в СК» и «Отправить в Альянс» работают как обычно;
- письмо отправляется, snapshot сохраняется, история пишется;
- статус `BillingTask` **не меняется автоматически** — задача остаётся в «Требуется запрос»;
- статусы меняются только через старую форму «Смена статуса» в левой колонке карточки или через bulk-update.

Это удобный режим на старте: один и тот же платёж можно прогонять много раз («отправить в СК», прийти проверить, что пришло, потом ещё раз отправить, потом ещё раз»), не возвращая статус через админку каждый раз.

Когда тестирование пройдено и пора включать production-логику — снимите флаг (`=True` или удалите переменную из `.env.prod`) и перезапустите `web` + `celery_worker`. После этого следующая успешная отправка будет переводить задачу вперёд по статусам.

Важно: «не меняет автоматически» означает только success-hook. Защита от отката (см. §S1 в техническом плане) и проверка «нельзя поставить уже отправленное письмо в очередь повторно» работают всегда, независимо от флага.

## 7. Что мониторить и где смотреть

### 7.1 В UI

- блок «История отправок» в карточке задачи — самый частый источник правды;
- `/admin/communications/outboundemail/` — список всех писем, фильтры по `kind`, `status`, `account`, datetime;
- `/admin/communications/emaildeliveryattempt/` — попытки отправки с ошибками и ответами провайдера.

### 7.2 В логах

```bash
docker-compose -f docker-compose.prod.yml logs --tail=200 -f celery_worker
docker-compose -f docker-compose.prod.yml logs --tail=200 -f web
```

Worker логирует:

- `Failed to send outbound email <id>` — провайдер вернул ошибку;
- `Failed to enqueue outbound email <id>` — Celery не смог поставить задачу в Redis.

Web-сервис логирует только создание/запросы; собственно отправку делает worker.

### 7.3 В Sentry (если подключён)

Все исключения из `send_outbound_email_now` и `queue_outbound_email` уходят в Sentry с traceback'ом. Полезный фильтр в Sentry: `event.module:apps.communications`.

## 8. Troubleshooting

### Письмо висит в статусе `queued`

- проверьте, что `celery_worker` запущен и не падает;
- посмотрите в Redis, нет ли затыка очередью;
- убедитесь, что `COMMUNICATIONS_EMAIL_ENABLED=True` и переменные SMTP заданы.

### Письмо переходит в `failed`, `last_error: Authentication failed`

- значит SMTP не принял логин/пароль;
- проверьте, что используете именно «пароль приложения», а не основной пароль;
- логин — полный email с доменом (`onlinepolis-billing@yandex.ru`, не `onlinepolis-billing`).

### Письмо переходит в `failed`, `last_error: 5.7.1 ... not allowed`

- Яндекс отверг отправку, потому что From-адрес не совпадает с авторизованным ящиком;
- выставьте `COMMUNICATIONS_FROM_EMAIL` равным `COMMUNICATIONS_SMTP_USERNAME`.

### Письмо отправилось, но попало в спам

- скорее всего, домен ящика не прошёл SPF/DKIM-проверку у получателя;
- для бесплатного `@yandex.ru` Яндекс сам ставит DKIM, но Gmail/Mail.ru могут не доверять «не свой» домен в From;
- для production переходите на корпоративный домен в Яндекс 360 и попросите администратора почты настроить SPF/DKIM/DMARC;
- проверьте текст письма на «спам-триггеры»: КАПС, слишком много восклицательных, ссылки на shortener'ы.

### `Не удалось поставить письмо в очередь отправки`

- ошибка соединения с Redis;
- проверьте `docker-compose -f docker-compose.prod.yml logs redis`;
- письмо сразу переводится в `failed`, нажмите «Повторить отправку» после починки Redis.

### Кнопка «Отправить в СК» не появляется у пользователя

Чек-лист по очереди:

1. Пользователь авторизован;
2. `COMMUNICATIONS_RESTRICT_TO_SUPERUSER` — если True, пользователь должен быть суперпользователем;
3. Если флаг False — у пользователя должен быть permission `communications.send_outbound_email` (через группу или напрямую);
4. После изменения permissions нужно перелогиниться или перезапустить `web` (Django кеширует permissions в сессии).

### Заметили лишний дубль письма

- если двойной клик прошёл быстрее, чем JS заблокировал кнопку (это редкость), помогает 60-секундное окно дедупликации;
- если письмо всё же ушло дважды (например, ручной POST через curl) — это видно в admin'е, оба письма с одинаковыми subject и разными ID; статус задачи изменится только один раз (последующие success-hook не откатывают вперёд).

### Хочу отменить ошибочно созданное письмо

Если письмо в статусе `draft` или `failed` — можно открыть его в админке и вручную перевести в `cancelled`. Из `queued`, `sending`, `sent` — отменить нельзя (письмо уже у получателя или вот-вот будет).

## 9. Лимиты Яндекса

- Обычный `@yandex.ru` ящик: примерно 500 писем в сутки, 30 писем в час, ~150 получателей суммарно за раз. Точные значения Яндекс не публикует.
- Яндекс 360 для бизнеса: лимиты выше (тарифные), но всё равно конечные.
- Если упёрлись — приходит ошибка `554 5.7.1 Message rejected under suspicion of SPAM` или `5.7.1 Send quota exceeded`.

При активной работе нескольких сотрудников 500 писем в день закончатся быстро. Для production переезжайте на Zimbra или Яндекс 360.

## 10. Откат на старый рабочий процесс

Если что-то пошло сильно не так и нужно временно отключить отправку:

1. Поставьте `COMMUNICATIONS_EMAIL_ENABLED=False` в `.env.prod`.
2. Перезапустите `web` и `celery_worker`.
3. Кнопки в UI у суперпользователя останутся видимыми, но сервер ответит «Отправка писем временно отключена». Пользователи продолжат работать через копирование текстов и ручную отправку — это и есть fallback-сценарий из плана.

Откат к коду: коммиты атомарные, можно сделать `git revert <commit>` без потери базы. Миграции `0001_initial` и `0002_alter_outboundemail_options` обратимы (Django сам сгенерирует `migrate communications zero`), но писать руками `manage.py migrate communications zero` стоит только если уверены, что таблицы и связанные `OutboundEmail` больше не нужны.

## 11. Что нужно подготовить к следующему этапу

После того как Яндекс-тест прошёл успешно, на этап 2 (четверговый отчёт через communications) и далее понадобятся:

- решение, какой ящик будет в production: Яндекс 360 или Zimbra;
- если Zimbra — SMTP host/port, логин, пароль системного ящика;
- список адресов СК и Альянса для будущего автоподбора (этап 3);
- решение, кто из менеджеров получает доступ к отправке после снятия `COMMUNICATIONS_RESTRICT_TO_SUPERUSER`.

Все эти решения зафиксируйте в `docs/improvement plans/email-communications-technical-plan.md` или в отдельной заметке — это сэкономит время следующей итерации.
