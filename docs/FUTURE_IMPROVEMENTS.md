## Активные задачи

### 1. Переход на полный производственный календарь РФ (isdayoff.ru)

**Контекст.** В `apps/core/date_utils.previous_business_day` сейчас
используется библиотека `holidays` (`holidays.RU`). Она знает все
фиксированные государственные праздники и часть перенесённых выходных,
но **может не знать про переносы рабочих суббот** и подвижные выходные,
которые Правительство РФ объявляет отдельным постановлением каждый
календарный год (например, объявление субботы рабочим днём в порядке
переноса с другого дня).

**Где используется.** Метод
`apps.billing.models.BillingTask.build_alliance_letter_text` — рассчёт
даты «Оплатить до» в письме Альянс-лизингу (на один рабочий день
раньше дедлайна СК). Любая ошибка в календаре приведёт к тому, что
лизингу будет указана нерабочая дата (либо, наоборот, не учтётся
рабочая суббота).

**Что нужно сделать.**

1. Завести единый источник производственного календаря РФ. Кандидаты:
   - **isdayoff.ru** — публичный бесплатный API
     (`https://isdayoff.ru/20260108` → `0` рабочий / `1` нерабочий).
     Учитывает официальный календарь Минтруда, включая переносы.
     Минусы: внешний сервис без SLA, требует кеша и фолбэка.
   - **Своя таблица в БД** (`WorkCalendar(date, is_workday)`).
     Заполняется один раз в год management-командой
     (например, из выгрузки isdayoff). Плюс: офлайн, видно в админке,
     можно править руками под форс-мажор.
2. Спрятать выбор источника за интерфейсом `date_utils.is_business_day`
   так, чтобы место вызова (`previous_business_day` и его потребители)
   не менялось.
3. Кеширование: использовать Redis (уже в стеке) с TTL не менее
   года — производственный календарь меняется только при выпуске
   нового постановления.
4. Фолбэк на текущую реализацию (`holidays.RU`) при недоступности
   внешнего источника, с записью события в логи.
5. Тесты: добавить кейс с заведомо известным переносом субботы
   (например, объявленным указом на конкретный год).

**Зачем сейчас не делаем.** В обычное время года `holidays.RU`
покрывает 95+% случаев без внешних зависимостей и без новой
инфраструктуры. Делать сразу более тяжёлое решение нет ROI, пока
не появятся реальные жалобы или инцидент с перенесённой субботой.

**Когда вернуться к задаче.**
- Если из Альянса/СК поступит замечание про неверный сдвиг даты.
- Если бухгалтерия укажет на пропущенный перенос.
- В декабре каждого года — проверить, не потребуется ли это
  на следующий календарный год.
