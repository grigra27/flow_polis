# План развития почтового контура и автоматизации писем

Дата фиксации обсуждения: 2026-05-06.

## 1. Зачем нужен этот документ

Этот документ фиксирует идею развития почтовой функциональности проекта: от текущего ручного копирования тем и текстов писем к полноценному почтовому контуру внутри системы.

Сейчас пользовательский сценарий в блоке "Очередные взносы" устроен так:

1. Сотрудник открывает задачу очередного взноса.
2. Система показывает тему и текст письма в страховую компанию.
3. Сотрудник копирует тему и текст.
4. Сотрудник вручную создает письмо в почтовом клиенте и отправляет его.
5. После отправки сотрудник возвращается в систему и меняет статус задачи.
6. После получения счета от страховой сотрудник вручную отправляет второе письмо в Альянс-лизинг с приложенным счетом.
7. После отправки второго письма сотрудник снова меняет статус задачи.

Идея развития: система должна уметь отправлять такие письма сама, из интерфейса проекта. В дальнейшем этот же механизм должен использоваться не только для очередных взносов, но и для других бизнес-писем, например для четвергового отчета и других будущих сценариев.

В более дальней перспективе система может получить входящий почтовый контур: сама мониторить входящие письма, находить ответы страховых компаний, подгружать счета, связывать их с задачами и готовить или отправлять следующее письмо в Альянс-лизинг.

## 2. Как я понял бизнес-задачу

Главная цель не в том, чтобы просто добавить кнопку "Отправить письмо" в текущий блок очередных взносов. Главная цель - создать основу для общего почтового слоя проекта.

Этот слой должен позволить:

- отправлять письма из разных модулей системы;
- показывать пользователю предварительный просмотр письма до отправки;
- хранить историю отправок;
- хранить адресатов, тему, текст и вложения именно в том виде, в котором письмо было отправлено;
- видеть ошибки отправки и повторять попытку;
- автоматически менять статусы бизнес-объектов только после успешной отправки;
- в будущем получать входящие письма и связывать их с объектами системы;
- в будущем обрабатывать вложения из входящих писем.

Первым потребителем этого механизма будет модуль "Очередные взносы". Вторым вероятным потребителем будет четверговый отчет, который сейчас выгружается в экспорт и отправляется сотрудником вручную.

## 3. Важное уточнение по почтовой инфраструктуре

В обсуждении было уточнено, что почтовый сервер компании - Zimbra, а не Microsoft 365. Поэтому Microsoft Graph и Outlook-интеграция не являются целевой архитектурой.

Outlook может оставаться почтовым клиентом пользователя, но серверная интеграция проекта должна смотреть не на Outlook, а на почтовый сервер. В нашем случае целевой сервер - Zimbra.

Для Zimbra естественная базовая архитектура:

- SMTP - для отправки исходящих писем;
- IMAP - для чтения входящих писем;
- Zimbra REST/SOAP API - потенциальная более глубокая интеграция на будущих этапах, если возможностей SMTP/IMAP будет недостаточно.

На первом этапе можно тестировать не на боевой Zimbra, а на бесплатном почтовом ящике, например Яндекс.Почте или Gmail. Это допустимо, если архитектура сделана через заменяемый provider-слой, а бизнес-логика не зависит от конкретного почтового провайдера.

Предпочтительный тестовый вариант - Яндекс.Почта, потому что она ближе к классическому SMTP/IMAP-сценарию. Gmail тоже возможен, но чаще требует более строгих настроек безопасности, паролей приложений или OAuth.

## 4. Текущее состояние проекта

В проекте уже есть хорошая основа для первого сценария.

Модуль `apps/billing` уже содержит сущность задачи очередного взноса `BillingTask`. У задачи есть статусы:

- `to_request` - требуется запросить счет;
- `requested` - счет запрошен у страховой компании;
- `sent_to_leasing` - счет передан в оплату в Альянс.

Также `BillingTask` уже умеет формировать:

- тему письма в страховую компанию;
- текст письма в страховую компанию;
- HTML-версию письма в страховую компанию;
- тему письма в Альянс-лизинг;
- текст письма в Альянс-лизинг;
- HTML-версию письма в Альянс-лизинг.

Сейчас шаблон детальной страницы задачи показывает эти тексты и дает кнопки копирования. То есть бизнес-текст письма уже в системе есть, но транспортной отправки еще нет.

В проекте также уже есть базовая Django email-конфигурация через переменные:

- `EMAIL_BACKEND`;
- `EMAIL_HOST`;
- `EMAIL_PORT`;
- `EMAIL_USE_TLS`;
- `EMAIL_HOST_USER`;
- `EMAIL_HOST_PASSWORD`.

По умолчанию используется console backend, то есть письма печатаются в консоль, а не уходят наружу.

Также в проекте есть Celery, что важно для правильной отправки писем: отправка не должна блокировать HTTP-запрос пользователя.

Важный пробел в текущей модели данных: у страховой компании сейчас есть поле `contacts` как ссылка на контактную информацию, но нет структурированного email-адреса для автоматической отправки. У менеджера лизинговой компании email уже есть. Для автоматизации нужно добавить нормальное хранение email-адресов получателей.

## 5. Целевая пользовательская логика

### 5.1 Очередные взносы: письмо в страховую компанию

Целевой сценарий:

1. Сотрудник открывает задачу очередного взноса.
2. Система показывает карточку письма в страховую компанию:
   - получатель;
   - копии;
   - тема;
   - тело письма;
   - предупреждения, если данных не хватает.
3. Сотрудник проверяет письмо.
4. Сотрудник нажимает "Отправить в СК".
5. Система создает исходящее письмо и ставит его в очередь.
6. Celery отправляет письмо через настроенный почтовый provider.
7. После успешной отправки:
   - письмо получает статус `sent`;
   - задача получает статус "Счет запрошен у СК";
   - в истории задачи появляется событие отправки.
8. Если отправка не удалась:
   - письмо получает статус `failed`;
   - пользователь видит ошибку;
   - статус задачи не меняется;
   - письмо можно повторно отправить после исправления проблемы.

Статус задачи должен меняться не по факту клика, а по факту успешной отправки.

### 5.2 Очередные взносы: письмо в Альянс-лизинг

Целевой сценарий:

1. Страховая компания присылает счет.
2. На первом этапе сотрудник вручную загружает счет в задачу.
3. Система показывает карточку письма в Альянс-лизинг:
   - получатель;
   - копии;
   - тема;
   - тело письма;
   - вложенный счет.
4. Если счет не приложен, кнопка отправки должна быть недоступна или должна показывать явную ошибку.
5. Сотрудник нажимает "Отправить в Альянс".
6. Система отправляет письмо через общий почтовый слой.
7. После успешной отправки:
   - письмо получает статус `sent`;
   - задача получает статус "Передан в оплату в Альянс";
   - в истории задачи появляется событие отправки.

Для письма в Альянс наличие вложения является бизнес-критичным требованием.

### 5.3 Четверговый отчет

Сейчас четверговый отчет выгружается из интерфейса, а сотрудник отправляет его сам.

Будущий сценарий:

1. Сотрудник открывает раздел отчета.
2. Система формирует файл отчета.
3. Система создает черновик исходящего письма с вложенным отчетом.
4. Сотрудник проверяет получателей, тему и вложение.
5. Сотрудник нажимает "Отправить".
6. Отправка проходит через тот же общий почтовый контур.
7. История отправки сохраняется в системе.

Важно: отчет не должен иметь свою отдельную почтовую реализацию. Он должен использовать тот же `communications`-слой, что и очередные взносы.

### 5.4 Дальнее будущее: входящие письма и счета

Перспективный сценарий:

1. Система отправляет письмо в страховую компанию.
2. В письме есть технический маркер, по которому можно связать ответ с задачей:
   - уникальный идентификатор в теме;
   - `Message-ID`;
   - `In-Reply-To` / `References`;
   - дополнительный служебный заголовок, если provider позволит;
   - возможно, видимый код вроде `OP-BILLING-12345`.
3. Страховая отвечает на письмо и прикладывает счет.
4. Система периодически проверяет входящие письма через IMAP или Zimbra API.
5. Новое письмо сохраняется во внутренний `InboundEmail`.
6. Система пытается сопоставить письмо с задачей:
   - по цепочке reply headers;
   - по техническому коду;
   - по номеру ДФА;
   - по номеру полиса;
   - по страховщику;
   - по сумме или дате платежа.
7. Вложения сохраняются как `InboundEmailAttachment`.
8. На первом автоматизированном этапе система предлагает пользователю подтвердить найденный счет.
9. После подтверждения система готовит письмо в Альянс.
10. В более зрелой версии система может сама отправить письмо в Альянс, если уверенность сопоставления высокая и включен режим автоотправки.

Полностью автоматическую цепочку "получили счет - сразу отправили в Альянс" не стоит включать первой. Сначала нужен полуавтоматический режим с подтверждением пользователя.

## 6. Рекомендуемая архитектура

Рекомендуемый подход - создать отдельное приложение, например:

```text
apps/communications
```

Возможные альтернативные названия:

- `apps/mailbox`;
- `apps/mail`;
- `apps/messaging`;
- `apps/communications`.

Из этих вариантов наиболее удачное название - `communications`, потому что оно не ограничивает модуль только email-ом. В будущем через него теоретически могут проходить и другие внешние коммуникации.

### 6.1 Граница ответственности

`apps/billing` должен знать бизнес-смысл письма: какое письмо нужно создать, по какой задаче, когда менять статус.

`apps/reports` должен знать, какой отчет сформировать и какой файл приложить.

`apps/communications` должен знать, как:

- хранить письмо;
- хранить получателей;
- хранить вложения;
- показывать статус отправки;
- отправлять письмо через provider;
- повторять отправку;
- принимать входящие письма;
- сопоставлять входящие письма с бизнес-объектами;
- хранить историю попыток и ошибок.

Бизнес-модули не должны напрямую работать с SMTP, IMAP, Яндексом, Gmail или Zimbra.

## 7. Основные модели данных

### 7.1 MailAccount

Системный почтовый аккаунт.

Примерные поля:

- `name` - понятное название, например "Тестовый Яндекс" или "Zimbra Billing";
- `email` - адрес отправителя;
- `display_name` - отображаемое имя;
- `provider` - например `smtp_imap`, `zimbra_api`;
- `is_active`;
- `is_default`;
- `smtp_host`;
- `smtp_port`;
- `smtp_use_tls`;
- `smtp_use_ssl`;
- `imap_host`;
- `imap_port`;
- `imap_use_ssl`;
- `username`;
- `secret_name` или ссылка на secret/env-переменную.

Пароль лучше не хранить открытым текстом в БД. Предпочтительно хранить его в `.env`, секретах сервера или отдельном secret-хранилище, а в БД держать только имя секрета.

### 7.2 OutboundEmail

Исходящее письмо.

Примерные поля:

- `account`;
- `status`;
- `kind` - тип письма, например `billing_insurer_request`, `billing_alliance_forward`, `thursday_report`;
- `subject`;
- `body_text`;
- `body_html`;
- `from_email`;
- `reply_to`;
- `created_by`;
- `sent_by`;
- `sent_at`;
- `queued_at`;
- `failed_at`;
- `last_error`;
- `provider_message_id`;
- `content_type`;
- `object_id`;
- `content_object`.

Связь через `content_type/object_id` позволит привязывать письма не только к `BillingTask`, но и к отчетам или будущим объектам.

Статусы:

- `draft` - черновик создан, еще не готов к отправке;
- `ready` - письмо готово, пользователь может отправить;
- `queued` - поставлено в очередь;
- `sending` - worker отправляет;
- `sent` - успешно отправлено;
- `failed` - отправка не удалась;
- `cancelled` - пользователь отменил черновик.

### 7.3 OutboundEmailRecipient

Получатель письма.

Примерные поля:

- `email`;
- `name`;
- `recipient_type` - `to`, `cc`, `bcc`.

Отдельная модель лучше, чем хранить список адресов одной строкой. Это упростит проверки, отображение и будущую аналитику.

### 7.4 OutboundEmailAttachment

Вложение исходящего письма.

Примерные поля:

- `email`;
- `file`;
- `filename`;
- `content_type`;
- `size`;
- `source_type` - загружено пользователем, сформировано отчетом, получено из входящего письма;
- `source_content_type`;
- `source_object_id`.

Для письма в Альянс счет должен быть обязательным вложением.

### 7.5 EmailDeliveryAttempt или MailEvent

История попыток и событий.

Примерные поля:

- `email`;
- `event_type`;
- `attempt_number`;
- `status_before`;
- `status_after`;
- `provider_response`;
- `error_message`;
- `created_by`;
- `created_at`.

Важно хранить историю не только успешных отправок, но и ошибок.

### 7.6 InboundEmail

Входящее письмо.

Примерные поля:

- `account`;
- `provider_uid`;
- `message_id`;
- `in_reply_to`;
- `references`;
- `subject`;
- `from_email`;
- `from_name`;
- `received_at`;
- `body_text`;
- `body_html`;
- `raw_headers`;
- `folder`;
- `status`;
- `matched_content_type`;
- `matched_object_id`;
- `match_confidence`.

Статусы:

- `new`;
- `matched`;
- `unmatched`;
- `needs_review`;
- `processed`;
- `ignored`.

### 7.7 InboundEmailAttachment

Вложения входящего письма.

Примерные поля:

- `email`;
- `file`;
- `filename`;
- `content_type`;
- `size`;
- `checksum`;
- `status`;
- `linked_content_type`;
- `linked_object_id`.

Для счетов важно считать checksum, чтобы избежать дублей.

## 8. Исходящий контур

### 8.1 Общий процесс

1. Бизнес-модуль вызывает сервис создания письма.
2. Сервис создает `OutboundEmail`.
3. Тема, тело, получатели и вложения фиксируются как snapshot.
4. Пользователь видит preview.
5. Пользователь подтверждает отправку.
6. Письмо переводится в `queued`.
7. Celery-задача берет письмо и отправляет его через provider.
8. Результат сохраняется в БД.
9. Бизнес-модуль получает callback или отдельный service hook и обновляет связанный объект.

### 8.2 Почему нужна очередь

Отправка письма может зависнуть или завершиться ошибкой:

- SMTP-сервер временно недоступен;
- неверный пароль;
- истек пароль приложения;
- адрес получателя некорректен;
- вложение слишком большое;
- почтовый сервер вернул rate limit;
- соединение оборвалось.

Если отправлять письмо прямо в HTTP-запросе, пользователь будет ждать, а результат может быть нестабильным. Поэтому отправка должна идти через Celery.

### 8.3 Повторная отправка

Для `failed` писем нужна кнопка "Повторить отправку".

Повторная отправка должна создавать новую попытку, но не должна терять исходный snapshot письма. Если пользователь хочет изменить тему, текст или вложение, лучше создать новую версию письма или вернуть письмо в draft-режим.

### 8.4 Отправленные письма и папка Sent

Источник истины должен быть в БД проекта. Не стоит полагаться только на папку "Отправленные" в почте.

Но если пользователям важно видеть письма в Zimbra/Yandex webmail, можно добавить второй шаг:

- отправить письмо через SMTP;
- затем сохранить копию письма в папку Sent через IMAP APPEND или API провайдера.

Это можно сделать не в первом этапе, а как отдельное улучшение.

## 9. Входящий контур

Входящий контур нужен для будущей автоматизации счетов.

### 9.1 Первый реалистичный уровень

1. Система отправляет письмо в СК с уникальным кодом задачи.
2. Worker периодически читает входящие письма системного ящика.
3. Новые письма сохраняются в БД.
4. Система показывает пользователю список входящих писем, которые удалось связать с задачами.
5. Пользователь подтверждает, что вложение действительно является счетом.
6. Счет прикрепляется к задаче.
7. Система готовит письмо в Альянс.

### 9.2 Более продвинутый уровень

Система сама:

- распознает счет во вложениях;
- проверяет страховщика, номер полиса, ДФА, сумму и дату;
- прикрепляет счет к задаче;
- переводит задачу в промежуточный статус "Счет получен";
- готовит письмо в Альянс.

### 9.3 Самый дальний уровень

Система автоматически отправляет письмо в Альянс без участия пользователя, если:

- входящее письмо уверенно связано с задачей;
- вложение прошло проверки;
- сумма совпала;
- страховщик совпал;
- задача находится в правильном статусе;
- включена настройка автоотправки для такого сценария.

Этот уровень лучше включать только после накопления практики и статистики по полуавтоматическому режиму.

## 10. Provider-слой

### 10.1 Главный принцип

Бизнес-логика не должна знать, через какой ящик и какого провайдера уходит письмо.

Нужен интерфейс примерно такого уровня:

```text
MailProvider
  send(outbound_email)
  fetch_new_messages(account, since)
  append_to_sent(outbound_email)
```

Возможные реализации:

```text
SmtpImapProvider
  - Яндекс
  - Gmail
  - Zimbra

ZimbraApiProvider
  - возможно позже, если понадобится более глубокая интеграция
```

### 10.2 Тестовый провайдер на первом этапе

На первом этапе можно использовать бесплатный почтовый ящик.

Предпочтительный вариант для тестирования - Яндекс.Почта:

- SMTP/IMAP-сценарий ближе к тому, что будет в Zimbra;
- проще проверить базовую отправку;
- можно использовать отдельный тестовый ящик;
- меньше риска затянуться в OAuth-специфику Gmail.

Gmail тоже возможен, но может оказаться менее удобным для MVP из-за более строгих требований безопасности.

### 10.3 Боевой провайдер

Боевой вариант - Zimbra:

- SMTP для отправки;
- IMAP для чтения входящих;
- Zimbra API позже, если понадобится.

Переход с тестового Яндекс/Gmail на Zimbra должен быть в основном заменой настроек, а не переписыванием бизнес-кода.

Примерный набор настроек:

```text
MAIL_PROVIDER=smtp_imap
MAIL_DEFAULT_ACCOUNT=billing

MAIL_BILLING_EMAIL=...
MAIL_BILLING_DISPLAY_NAME=...

MAIL_BILLING_SMTP_HOST=...
MAIL_BILLING_SMTP_PORT=465
MAIL_BILLING_SMTP_USE_SSL=true
MAIL_BILLING_SMTP_USE_TLS=false

MAIL_BILLING_IMAP_HOST=...
MAIL_BILLING_IMAP_PORT=993
MAIL_BILLING_IMAP_USE_SSL=true

MAIL_BILLING_USERNAME=...
MAIL_BILLING_PASSWORD=...
```

Конкретные host/port для Zimbra нужно подтвердить у администратора вашей почты.

## 11. Варианты реализации

### 11.1 Самый простой вариант

Сделать отправку прямо в `apps/billing` через Django `EmailMessage` и текущие email settings.

Плюсы:

- быстрее всего;
- мало новых моделей;
- можно быстро проверить идею.

Минусы:

- решение будет заточено под очередные взносы;
- четверговый отчет придется делать отдельно или переносить код;
- входящие письма потом будет сложно добавить аккуратно;
- история отправок будет слабой;
- провайдерная логика смешается с бизнес-логикой;
- будет сложнее контролировать вложения и повторные отправки.

Такой вариант допустим только как временный прототип.

### 11.2 Оптимальный вариант

Сразу создать `apps/communications` и реализовать общий исходящий контур.

Плюсы:

- подходит для очередных взносов;
- подходит для четвергового отчета;
- можно расширять на другие письма;
- можно добавить входящие письма без переделки основы;
- провайдер можно заменить с Яндекса/Gmail на Zimbra;
- появляется нормальная история и аудит.

Минусы:

- больше начальной работы;
- нужно продумать модели и статусы;
- нужно сделать UI для preview и истории.

Это рекомендуемый вариант.

### 11.3 Полная целевая архитектура

Сразу реализовать:

- исходящие письма;
- входящие письма;
- IMAP-мониторинг;
- автоматическое сопоставление;
- обработку вложений;
- автоотправку второго письма.

Плюсы:

- сразу закрывает всю будущую картину.

Минусы:

- слишком большой объем для первого этапа;
- много неопределенности по качеству входящих писем от страховых;
- высокий риск сделать сложную систему до накопления реальных данных.

Не рекомендуется как первый этап.

## 12. Рекомендуемый план реализации

### 12.1 Зафиксированные решения для первого MVP

По итогам дополнительного обсуждения первый технический MVP должен быть уже и осторожнее, чем полная целевая архитектура.

Решения для MVP:

- первый релиз делается только для блока "Очередные взносы";
- четверговый отчет остается следующим этапом и не входит в первый релиз;
- отправка выполняется от общего системного почтового ящика;
- на этапе тестирования кнопка отправки доступна только суперпользователю;
- обычные пользователи продолжают видеть и использовать текущие кнопки копирования темы и текста;
- тема и тело письма не редактируются пользователем перед отправкой;
- адрес получателя вводится вручную в интерфейсе отправки;
- автоматический выбор адресов страховых компаний, Альянса или менеджеров не входит в первый MVP;
- кнопки копирования остаются как fallback и как основной рабочий сценарий для менеджеров до завершения тестирования;
- копию письма в папку Sent почтового ящика на первом этапе не сохраняем;
- источник истины по отправкам - база проекта;
- после успешного тестирования доступ к отправке можно будет открыть всем авторизованным пользователям, у которых есть доступ к разделу;
- разрешенные форматы вложений для счета: `pdf`, `jpg`, `jpeg`, `png`, `doc`, `docx`, `xls`, `xlsx`;
- рекомендуемый лимит вложения для старта: 10-15 MB на файл.

Особое решение по техническому коду сопоставления: для MVP не обязательно добавлять код в тему письма. Более мягкий вариант - добавлять код в служебный нижний блок тела письма и параллельно сохранять технические email headers. Это оставляет тему письма чистой, но дает будущему входящему контуру несколько способов связать ответ с исходной задачей.

Пример служебной строки в теле:

```text
Код запроса: OP-BILLING-12345
```

Пример headers, которые может добавить система при отправке:

```text
X-Onlinepolis-Object: billing-task-12345
X-Onlinepolis-Email-Kind: billing_insurer_request
```

Важно понимать ограничение: при ответе страховой компании служебные headers исходного письма обычно не возвращаются в новом входящем письме. Надежнее всего для будущего сопоставления хранить `Message-ID` исходящего письма и искать ответы по `In-Reply-To` / `References`. Видимый код в теле письма полезен как запасной способ, особенно если письмо пересылают или отвечают нестандартно.

### Этап 0. Решения до разработки

Нужно определить:

- какой ящик будет использоваться для тестирования: вероятно, Яндекс;
- какой ящик будет использоваться в бою: Zimbra;
- будет ли отправка от общего ящика или от личных ящиков сотрудников;
- какие адреса СК нужны;
- куда отправлять письма в Альянс;
- нужны ли копии руководителям, филиалам или общему архивному ящику;
- нужно ли сохранять копии в папке Sent;
- какие типы файлов разрешены для счетов;
- какой максимальный размер вложений;
- кто имеет право отправлять письма;
- нужен ли режим "только подготовить черновик, но не отправлять".

Рекомендуемое решение для старта:

- использовать общий тестовый Яндекс-ящик на staging;
- использовать общий Zimbra-ящик в production;
- личные ящики сотрудников не подключать на первом этапе;
- источник истины по отправкам хранить в БД проекта;
- копии в Sent сделать опциональным улучшением;
- отправку на время тестирования показывать только суперпользователю;
- адрес получателя на первом этапе вводить вручную;
- тему и текст письма показывать без возможности редактирования.

### Этап 1. Общий исходящий контур

Создать `apps/communications`.

Реализовать:

- `MailAccount`;
- `OutboundEmail`;
- `OutboundEmailRecipient`;
- `OutboundEmailAttachment`;
- `EmailDeliveryAttempt` или `MailEvent`;
- provider-интерфейс;
- `SmtpImapProvider.send`;
- Celery-задачу отправки;
- admin или простую служебную страницу для просмотра писем;
- retry failed-писем.

На этом этапе можно отправлять письма на тестовые адреса.

### Этап 2. Интеграция с очередными взносами

Добавить в `apps/billing` сервисы:

- создать письмо запроса счета в СК;
- создать письмо отправки счета в Альянс;
- проверить обязательные данные;
- после успешной отправки обновить статус задачи.

Обновить интерфейс задачи:

- оставить кнопки копирования как fallback;
- добавить кнопку подготовки письма;
- добавить preview письма;
- добавить кнопку отправки;
- показывать кнопку отправки только суперпользователю на этапе тестирования;
- добавить поле ручного ввода email-адреса получателя;
- добавить блок истории писем;
- добавить загрузку счета для письма в Альянс.

Важно: для письма в Альянс отправка должна быть невозможна без вложения счета.

### Этап 3. Адреса получателей

Добавить структурированное хранение email-адресов.

Минимально:

- email или список email-ов для страховой компании;
- email для Альянса;
- cc/bcc-настройки.

Лучше:

- модель `CommunicationContact`;
- связь с `Insurer`, `Branch`, `LeasingManager`;
- тип контакта: счет, общий, менеджер, копия, архив;
- активность контакта;
- комментарий.

Не стоит парсить email-ы из текстового поля или ссылки на контакты. Для автоматической отправки нужны структурированные и проверяемые данные.

### Этап 4. Четверговый отчет

Интегрировать отчет с `apps/communications`.

Сценарий:

- отчет формируется как файл;
- создается `OutboundEmail`;
- файл добавляется как attachment;
- пользователь проверяет письмо;
- письмо отправляется через общий контур.

Этот этап должен переиспользовать уже готовые модели и provider, без отдельной реализации отправки.

### Этап 5. Входящий контур

Добавить:

- `InboundEmail`;
- `InboundEmailAttachment`;
- IMAP polling Celery-задачу;
- дедупликацию входящих писем;
- сохранение вложений;
- первичное сопоставление с задачами;
- экран входящих писем, требующих проверки.

На этом этапе система еще не должна автоматически отправлять письма в Альянс. Она должна помогать пользователю найти и привязать счет.

### Этап 6. Полуавтоматическая обработка счетов

Добавить:

- правила сопоставления;
- оценку уверенности;
- проверку номера полиса и ДФА;
- проверку страховщика;
- проверку суммы, если ее можно извлечь;
- привязку вложения к `BillingTask`;
- автосоздание черновика письма в Альянс.

Пользователь подтверждает отправку.

### Этап 7. Осторожная автоотправка

Добавить автоотправку только для сценариев с высокой уверенностью.

Нужны ограничения:

- включается настройкой;
- работает только для определенных страховщиков или типов задач;
- не работает при неоднозначных данных;
- не работает при нескольких счетах;
- не работает при несовпадении суммы;
- всегда пишет подробную историю.

## 13. Технические нюансы

### 13.1 Идемпотентность

Нельзя случайно отправить одно и то же письмо дважды из-за повторного клика или повторного запуска Celery.

Нужно:

- блокировать повторную отправку писем в статусах `queued`, `sending`, `sent`;
- использовать транзакции;
- возможно, использовать `select_for_update`;
- хранить отдельные delivery attempts.

### 13.2 Ошибки отправки

Не использовать `fail_silently=True` для бизнес-писем.

Ошибки должны:

- сохраняться в БД;
- показываться пользователю;
- попадать в логи;
- не менять бизнес-статус задачи на успешный.

### 13.3 Безопасность секретов

Пароли от почтовых ящиков нельзя хранить в открытом виде в репозитории.

Предпочтительно:

- `.env` для локальной разработки;
- production secrets для боевого окружения;
- в БД только ссылка на secret или имя аккаунта.

### 13.4 Права доступа

Нужны права:

- кто может создавать черновики;
- кто может отправлять письма;
- кто может повторять отправку;
- кто может менять получателей;
- кто может просматривать историю писем;
- кто может открывать вложения.

Для начала можно завязаться на существующую авторизацию и потом расширить роли.

### 13.5 Вложения

Для счетов и отчетов нужно контролировать:

- допустимые типы файлов;
- максимальный размер;
- имя файла;
- хранение файла;
- повторное использование файла;
- источник файла;
- удаление или архивирование.

Для входящих вложений в будущем желательно считать checksum.

### 13.6 SPF, DKIM, DMARC и доставляемость

Для боевого Zimbra-ящика важно, чтобы домен был корректно настроен:

- SPF;
- DKIM;
- DMARC;
- корректный From;
- корректный Reply-To.

Иначе письма могут попадать в спам или отклоняться получателями.

### 13.7 Общий ящик или личные ящики

Для первого этапа лучше общий ящик.

Плюсы общего ящика:

- проще настроить;
- проще аудит;
- проще входящий мониторинг;
- ответы приходят в одно место;
- не нужно хранить токены или пароли каждого сотрудника.

Минусы:

- письмо отправляется не от конкретного сотрудника;
- может потребоваться указать сотрудника в подписи.

Личные ящики сотрудников можно рассмотреть позже, если бизнесу это действительно нужно.

### 13.8 Подписи

Нужно решить, где формируется подпись:

- в тексте письма внутри системы;
- на уровне шаблона письма;
- на уровне почтового сервера.

Если отправка идет через SMTP, автоматическая подпись почтового клиента не добавится. Значит подпись лучше формировать внутри системы.

### 13.9 Ответы на письма

Для будущей автоматизации входящих писем нужно задавать `Reply-To` так, чтобы ответы приходили в мониторируемый ящик.

Если письмо отправлено с общего ящика, это просто.

Если письмо отправлено с личного ящика, входящий мониторинг усложняется.

### 13.10 Маркеры сопоставления

В письма желательно добавлять технический маркер.

Варианты:

- видимый код в теме: `OP-BILLING-12345`;
- скрытый заголовок `X-Onlinepolis-Object`;
- уникальный `Message-ID`;
- текстовый код внизу письма.

Для первого MVP предпочтительно не менять тему письма, а использовать комбинацию:

- технический код внизу тела письма;
- сохраненный `Message-ID` исходящего письма;
- служебные headers исходящего письма.

Видимый код в теме может быть надежнее при нестандартных пересылках, потому что тема обычно сохраняется в цепочке писем. Но он сильнее влияет на внешний вид письма, поэтому его лучше оставить как запасной вариант на случай, если сопоставление по `Message-ID`, `In-Reply-To`, `References` и коду в теле окажется недостаточно надежным.

## 16. Что важно протестировать

Для исходящей отправки:

- письмо создается как draft;
- письмо нельзя отправить без получателя;
- письмо нельзя отправить без темы;
- письмо в Альянс нельзя отправить без счета;
- успешная отправка меняет статус письма;
- успешная отправка меняет статус `BillingTask`;
- ошибка отправки не меняет статус `BillingTask`;
- failed-письмо можно повторить;
- sent-письмо нельзя отправить повторно случайным кликом;
- история попыток сохраняется.

Для вложений:

- файл прикладывается к письму;
- файл уходит через SMTP;
- большой файл корректно отклоняется или обрабатывается;
- недопустимый тип файла не принимается.

Для входящих писем в будущем:

- письмо сохраняется один раз;
- повторный polling не создает дубликаты;
- вложения сохраняются;
- письмо связывается с задачей по `In-Reply-To`;
- письмо связывается с задачей по коду в теле или, если позже включим такой формат, по коду в теме;
- неоднозначное письмо уходит в ручную проверку.

## 17. Нюансы по текущему UI

Кнопки "Скопировать тему" и "Скопировать текст" лучше не удалять сразу.

На первом этапе они могут оставаться fallback-сценарием:

- если почта временно не работает;
- если нужно отправить нестандартное письмо;
- если пользователь хочет отправить вручную;
- если тестирование еще не завершено.

Новый UI можно добавить рядом:

- "Подготовить письмо";
- поле ручного ввода email-адреса получателя;
- "Отправить";
- "Повторить";
- "Загрузить счет";
- "История писем";
- "Отправлено вне системы".

На время тестирования кнопку "Отправить" видит только суперпользователь. Для остальных пользователей основной рабочий сценарий остается прежним: скопировать тему и текст, отправить письмо вручную через почтовый клиент и затем обновить статус задачи.

Кнопка "Отправлено вне системы" нужна для случаев, когда сотрудник все равно отправил письмо из почтового клиента. Тогда можно корректно обновить статус задачи и историю, не создавая ложную запись о том, что письмо отправила система.

## 18. Итоговая рекомендация

Оптимальный путь:

1. Не делать узкую отправку только внутри `billing`.
2. Создать отдельное приложение `apps/communications`.
3. Сначала реализовать общий исходящий outbox.
4. В качестве первого provider использовать SMTP/IMAP.
5. На staging тестировать на Яндекс.Почте или другом бесплатном ящике.
6. В production переключиться на Zimbra.
7. Первым бизнес-сценарием подключить "Очередные взносы".
8. Вторым бизнес-сценарием подключить четверговый отчет.
9. Потом добавить входящий IMAP-мониторинг.
10. Потом добавить полуавтоматическую обработку счетов.
11. Только после этого рассматривать осторожную автоотправку второго письма в Альянс.

Такой подход дает быстрый первый результат, но не превращает первый этап в временный код, который придется выбрасывать.
