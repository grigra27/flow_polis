# Возможный публичный API для выгрузки полиса в JSON

## Назначение

Документ описывает возможный формат публичной или партнерской JSON-выгрузки по одному страховому полису. Цель формата - отдавать карточку полиса в виде стабильного контракта, который не повторяет Excel-отчеты один в один, а отражает текущую доменную модель проекта:

- `Policy` хранит шапку договора, стороны, объект страхования, статусы и дополнительные поля.
- `PaymentSchedule` хранит график платежей, страховую сумму, премию, комиссии и платежные статусы.
- Справочники `Client`, `Insurer`, `Branch`, `InsuranceType`, `LeasingManager`, `CommissionRate` раскрываются внутри ответа минимальными объектами.

JSON Schema и пример ответа лежат рядом: `docs/public_api/policy_v1_schema_and_example.json`.

## Почему формат вложенный

Для JSON лучше не использовать плоскую структуру Excel-колонок. В текущей базе страховая сумма и премия находятся на уровне платежей, а не самого полиса. Поэтому корректная карточка полиса должна содержать:

- шапку полиса;
- стороны договора;
- объект страхования;
- финансовую сводку;
- полный график платежей;
- дополнительные метки и служебные поля.

Такой формат устойчивее для интеграций: внешний потребитель видит первичные данные платежей и отдельно агрегаты, которые можно перепроверить.

## Возможный endpoint

```http
GET /api/v1/policies/{id}/
```

Опциональные параметры:

- `profile=public` - безопасный внешний профиль по умолчанию.
- `profile=internal` - расширенный профиль для внутренних интеграций.
- `include=billing` - в будущем может добавлять задачи очередных взносов из `BillingTask`.

Пример:

```http
GET /api/v1/policies/123/?profile=public
```

## Общая структура ответа

```json
{
  "schema_version": "policy.v1",
  "exported_at": "2026-05-26T16:40:00+03:00",
  "policy": {}
}
```

`schema_version` нужна, чтобы можно было менять контракт без поломки потребителей. Для несовместимых изменений лучше вводить `policy.v2`, а не менять смысл существующих полей.

## Основные разделы `policy`

| Раздел | Смысл |
| --- | --- |
| `id`, `policy_number`, `dfa_number` | Идентификаторы полиса в системе и бизнес-номера |
| `period` | Даты действия договора и дата расторжения |
| `statuses` | Активность полиса, ДФА, загрузка полиса, участие брокера |
| `parties` | Лизингополучатель, страхователь, страховщик, филиал, менеджер |
| `insurance` | Вид страхования, объект, франшиза |
| `finance_summary` | Агрегаты: общая премия, комиссия, тарифы по годам |
| `payments` | Первичный график платежей |
| `info` | Инфо-метки и свободные поля |
| `timestamps` | Системные даты создания и изменения |

## Деньги и точность

Денежные значения предлагается отдавать объектом:

```json
{
  "amount": "150000.00",
  "currency": "RUB"
}
```

`amount` хранится строкой, а не числом JSON, чтобы не терять копейки из-за float-округления у потребителей API.

## Даты

Даты без времени отдаются в ISO-формате:

```json
"2024-01-15"
```

Дата формирования ответа отдается как `date-time`:

```json
"2026-05-26T16:40:00+03:00"
```

## Статусы платежей

Поле `payments[].status` можно вычислять по текущей логике модели:

- `approved` - заполнена дата согласования акта с СК (`insurer_date`).
- `paid` - заполнена фактическая дата оплаты (`paid_date`), но акт еще не согласован.
- `cancelled` - платеж отменен из-за расторжения полиса.
- `overdue` - платеж просрочен и не оплачен.
- `waiting` - платеж ожидается.

Это соответствует тому, как статусы уже отображаются в карточке полиса.

## Профили доступа

### `public`

Минимальный профиль для внешней передачи:

- можно отдавать номера, даты, стороны, объект, платежи, финансовые суммы;
- лучше не отдавать внутренние комментарии страховщика, email страховщика, `info4`, служебные заметки справочников;
- `info4` можно возвращать как `null` или полностью исключать, если endpoint будет поддерживать разные представления.

### `internal`

Профиль для внутренних систем:

- может включать `info4`, заметки, контакты, email, ссылки на карточки в системе;
- может включать будущие данные по `BillingTask`: статус запроса счета, ответственный, дедлайны, история действий.

## Техническая реализация

На первом этапе достаточно отдельного сериализатора без Django REST Framework:

- `apps/policies/serializers.py` - функции `serialize_policy(policy, profile="public")`, `serialize_payment(payment)`, `money(value)`.
- `apps/policies/views.py` - `PolicyJsonDetailView` или function-based view с `JsonResponse`.
- `apps/policies/urls.py` - маршрут `api/v1/policies/<int:pk>/`.

Для производительности queryset должен заранее подтянуть связи:

- `select_related("client", "policyholder", "insurer", "branch", "insurance_type", "leasing_manager")`;
- `prefetch_related("payment_schedule__commission_rate", "info_tags__tag")`;
- для списковых API использовать `annotate`/`Subquery` для `premium_total`, как уже сделано в экспортерах.

## Версионирование

Рекомендуемый подход:

- `policy.v1` - первая стабильная схема карточки полиса.
- Новые необязательные поля можно добавлять в `v1`.
- Переименование полей, изменение типов или удаление полей - только через `policy.v2`.

## Открытые вопросы

- Какой внешний идентификатор считать главным: `id`, `policy_number`, `dfa_number` или отдельный UUID.
- Нужно ли скрывать ИНН в публичном профиле.
- Нужны ли ссылки на документы полиса, если появится файловое хранилище полисов.
- Нужен ли отдельный списковый endpoint для массовой выгрузки: `GET /api/v1/policies/?updated_from=...`.
