Как формируется четверговый отчет

Назад к экспортам
Кратко (TL;DR)
  • В отчет попадают полисы с неподгруженными документами и/или неоплаченными платежами.
  • Для неоплаченных платежей учитываются только платежи с датой по договору до выбранной даты.
  • Если у полиса обе причины, в файле будет одна строка с двумя пунктами в поле «Причина».
Параметры запуска
Параметр Где задается Как влияет
payment_date Поле «Дата для раздела 2» Ограничивает раздел «Нет данных об оплате»: берутся платежи с due_date <= payment_date.
payment_date не указан По умолчанию Используется текущая дата сервера.
branch (технический) Query-параметр URL Если передан, ограничивает полисы конкретным филиалом.
Как данные попадают в отчет
  1. Берутся полисы, у которых признак «Полис подгружен» = Нет.
  2. Берутся платежи, у которых «Дата факт. оплаты» пустая, а полис активен.
  3. Для платежей применяется фильтр по дате: только до выбранной даты включительно.
  4. Записи объединяются по филиалу и полису, чтобы убрать дубли.
  5. Для каждого полиса формируется одна строка: номер полиса/ДФА, участники, даты, объект, сумма и причина.
  6. Поле «Причина» заполняется по факту попадания в разделы:
    • не подгружены документы
    • нет данных об оплате
Что означают ключевые столбцы
  • Очередной взнос / Дата платежа по договору / Дата факт. оплаты: показываются из первого неоплаченного платежа (с учетом выбранной даты); если таких нет, берется первый платеж по графику.
  • Причина: может содержать одну или две причины; если две, они выводятся в одной ячейке через перенос строки.
  • Группировка: в файле данные идут блоками по филиалам.
FAQ

Полис не попадет в отчет, если у него подгружены документы и при этом нет неоплаченных платежей, подходящих под выбранную дату.

Неоплаченным считается платеж, у которого поле «Дата факт. оплаты» пустое.

В раздел «Нет данных об оплате» попадают только неоплаченные платежи по активным полисам. Это исключает технический шум от неактивных договоров.

Эта дата ограничивает платежи по договорной дате. Попадут только платежи с датой по договору не позже выбранной даты.

Это значит, что одновременно не подгружены документы и есть неоплаченный платеж, подходящий по дате. В отчете такие причины объединяются в одной строке.

Система сначала пытается показать первый неоплаченный платеж (с учетом даты фильтра). Если такого нет, берется первый платеж по графику для информативности строки.

Да, в backend поддерживается фильтр по филиалу через query-параметр branch. На текущей форме этот фильтр не вынесен отдельным полем.
